# 鼎泰上下料项目软件规划

## 1. 项目目标

本项目面向设备上下料场景，目标是在保证安全与稳定的前提下，实现机器人自主完成料盒更换流程，并支持异常检测、故障恢复与可追溯日志。控制系统遵循“安全先于动作”原则：任何控制 API 在实际下发前都必须通过统一安全检查。

主流程如下：

`控制机器人本体接近目标设备` -> `双手开门` -> `识别料盒位置` -> `左手控制松夹` -> `左手取出料盒` -> `放入腹部的料仓中` -> `左手取出空料盒` -> `放入设备中` -> `夹紧料盒` -> `双手关门`

## 2. 硬件与传感器配置

- 机器人本体：AGV 小车、升降轴、两只机械臂、灵巧手（左手）、三指手（右手）
- 传感器：RGBD 相机、IMU
- 预留扩展：门状态检测、夹爪力反馈、急停输入、安全光栅（如现场接入）

## 3. 软件架构

### 3.1 分层架构

```ultree
├─ 应用层（任务编排 / 状态机 / 作业管理）
│   ├─ 规划层（抓取点计算 / 路径规划 / 动作序列生成）
│   ├─ 执行层（机械臂控制 / 夹爪控制 / AGV 控制 / 升降轴控制）
│   └─ 设备通信层（机器人控制器 / PLC / 相机 SDK / IO）
└─ 横切模块：安全监控、日志追踪、配置管理、告警上报
```

### 3.2 进程与线程建议

- 主控制进程：运行任务状态机与调度逻辑
- 感知进程：图像采集、点云处理、位姿识别
- 安全监控线程：急停、碰撞、超时、门状态互锁
- 通信线程池：与机器人、PLC、相机进行异步收发

### 3.3 坐标系约定

- `base`：机器人基座坐标系
- `camera`：相机坐标系
- `left_hand`：左手坐标系
- `right_hand`：右手坐标系  
- `bunker`：腹部料仓坐标系
- `device`：目标设备坐标系

坐标转换链需统一配置管理，不允许在业务代码中分散硬编码。

## 4. 功能模块划分

```ultree
.
├─ 工作上下文
│  ├─ 任务状态机
│  ├─ 轨迹规划层
│  ├─ 运动控制层
│  ├─ 传感器层
│  └─ 安全层
├─ 通信层
└─ 识别算法层
```

### 4.1 任务状态机模块

- 负责流程编排、步骤切换、异常回退、重试策略
- 对外提供：`start_job`、`stop_job`、`pause_job`、`resume_job`

### 4.2 感知与识别模块

- RGBD 数据采集与预处理
- 料盒位姿识别与置信度评估
- 识别失败重拍与降级策略

### 4.3 规划与控制模块

- 抓取姿态计算、避障路径规划
- 左右手协同动作与互锁
- 夹持力、松夹时序与放置位控制
- 所有控制类 API 必须通过统一安全管理装饰器，执行前做安全前置检查

### 4.4 通信模块

- 统一封装机器人控制器、相机 SDK、PLC/IO 接口
- 提供心跳、超时、重连、错误码映射

### 4.5 安全模块

- 急停监控与状态广播
- 门状态互锁与危险区动作限制
- 任务超时、动作失败、碰撞检测后的安全退避

### 4.6 控制上下文统一能力（强制）

- 每个控制上下文（机械臂、夹爪、AGV、升降轴）都必须提供运动状态轮询或订阅监控能力
- 每个运动类指令必须提供“最终到位”确认方法，至少包含：超时、误差阈值、稳定时间窗
- 指令完成语义统一为：`指令下发成功 != 运动完成`，只有到位确认通过才算成功

## 5. 流程状态机定义（建议）

### 5.1 主状态

- `IDLE`：待机
- `APPROACH_DEVICE`：本体接近设备
- `OPEN_DOOR`：双手开门
- `DETECT_BOX`：识别料盒
- `UNCLAMP_BOX`：松夹
- `TAKE_FULL_BOX`：取出满料盒
- `PUT_TO_BIN`：放入腹部料仓
- `TAKE_EMPTY_BOX`：取空料盒
- `PLACE_TO_DEVICE`：放回设备
- `CLAMP_BOX`：夹紧料盒
- `CLOSE_DOOR`：双手关门
- `DONE`：完成
- `ERROR`：异常
- `SAFE_STOP`：安全停机

### 5.2 状态通用字段

- 进入条件
- 执行动作
- 成功条件
- 失败条件
- 超时阈值
- 重试上限
- 回退目标状态

## 6. 接口契约与数据约定

### 6.1 指令接口（示例）

- 运动控制：`move_to(pose, speed, accel, timeout)`
- 夹爪控制：`grip(force, width, timeout)` / `release(width, timeout)`
- 识别接口：`detect_box(frame) -> pose, score, timestamp`
- 任务接口：`submit_job(job_id, params)` / `query_job(job_id)`

#### 6.1.1 控制 API 安全装饰器约定（强制项）

- 控制相关接口统一使用安全管理装饰器（示例：`@control_safety_guard(...)`）
- 装饰器在方法执行前至少检查：
  - 急停状态、系统模式（Auto/Manual）、互锁条件（门/夹具/安全区）
  - 控制器在线状态、心跳状态、任务上下文状态（是否允许执行动作）
  - 目标参数合法性（位置/速度/加速度/力矩边界、软限位、禁行区）
- 任一检查失败，必须拒绝下发控制指令并记录结构化错误日志（含 `job_id`、`state`、`error_code`、`reason`）

#### 6.1.2 控制上下文运动监控接口约定（强制项）

- 每个控制上下文必须提供：
  - `query_motion_state() -> MotionState`
  - `wait_motion_done(timeout, tolerance, stable_window) -> MotionResult`
  - `monitor_motion_state(callback | queue)`（轮询或订阅二选一，推荐同时支持）
- `wait_motion_done` 必须用于所有运动类 API 的收尾确认，禁止仅以“控制器已接收指令”作为成功判定

### 6.2 数据约束

- 统一 SI 单位：位置 `mm`、角度 `deg`、时间 `s`
- 时间戳统一使用 UTC 或明确时区字段
- 全链路日志必须包含 `job_id`、`state`、`device_id`、`error_code`

## 7. 异常处理与恢复策略

- 识别失败：重拍最多 `N` 次，仍失败则进入 `ERROR`
- 抓取失败：退回安全位后重试最多 `M` 次
- 通信中断：进入降级态并重连，超时未恢复则 `SAFE_STOP`
- 安全触发（急停/碰撞）：立即中止当前动作并执行安全退避
- 运动未到位/到位超时：进入 `ERROR` 或 `SAFE_STOP`，并附带末次位置误差、轨迹段 ID、控制器状态快照

`N`、`M`、超时阈值需在配置文件中可调，不允许硬编码在业务逻辑中。

## 8. 日志、监控与可追溯

- 日志分级：`DEBUG`、`INFO`、`WARN`、`ERROR`
- 每次作业输出完整轨迹：状态转移、关键位姿、识别分数、控制指令回执
- 关键告警实时上报：识别连续失败、夹取失败、通信中断、安全触发

## 9. 配置与部署

- 配置分层：设备参数、任务参数、安全参数、算法参数
- 配置文件版本化管理，启动时输出配置摘要与哈希
- 支持仿真模式与实机模式，禁止混用设备端点

## 10. 验收指标与测试计划

### 10.1 验收指标（初版）

- 上下料成功率 >= 95%（连续样本窗口定义需明确）
- 单次作业节拍满足产线目标值（由现场给定）
- 关键位姿误差满足工艺阈值（由标定结果与工艺定义）
- 异常恢复平均时长满足运维目标值

### 10.2 测试计划

- 单元测试：状态机转换、接口参数校验、异常分支
- 集成测试：相机识别 + 规划 + 控制链路联调
- 联机冒烟：开门/取盒/放盒/关门完整流程
- 异常注入：断连、漏检、夹持失败、超时、急停触发

未接入真实硬件前，仅可声明软件级验证通过，不可声明硬件行为已验证。

## 11. 里程碑建议

- M1：完成状态机骨架与通信抽象，跑通仿真闭环
- M2：完成感知链路与手眼标定，打通单臂抓取
- M3：完成双臂协同开关门与完整上下料
- M4：完成稳定性优化、异常恢复、现场验收

## 12. 启动前数学准备（已落地）

### 12.1 纯数学手眼标定模块

- 模块路径：`src/calibration/hand_eye.py`
- 核心方程：`A_k X = X B_k`
- 能力：
  - 从同步位姿序列构造相对运动对（支持相邻帧/全组合）
  - 纯数学闭式求解 `AX=XB`（旋转 + 平移）
  - 输出标定残差（旋转 RMSE、平移 RMSE、最大误差）
  - 生成合成数据做离线自检（不依赖硬件）

### 12.2 启动前建议流程

1. 先用 `debug/test_handeye_math.py` 跑通算法与误差统计。
2. 再接入真实采样数据（机器人位姿序列 + 视觉位姿序列）验证结果分布。
3. 达到工艺阈值后，再进入感知与抓取联调，避免把数学问题和系统问题混在一起排查。
