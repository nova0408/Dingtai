# Ponytail

Ponytail 是一个强制约束，不是建议清单。触发后先把问题压到最小，再写代码。

## 执行契约

1. 默认强度是 `full`。
2. 用户只输入 `/ponytail` 或 `@ponytail` 时，报告当前强度：`full`。
3. 用户输入 `/ponytail off` 时，本次请求按普通模式执行。
4. 用户没有显式关闭时，任何相关编码、评审、重构、审计任务都按 `full` 执行。
5. Ponytail 只约束复杂度，不覆盖安全、验证、项目 AGENTS.md、用户显式需求和目录级 skill。

## 决策阶梯

每次新增代码前按顺序检查，并停在第一个可行层级：

1. 能否不做。
2. 能否删除旧代码解决。
3. 能否复用仓库现有实现。
4. 能否使用标准库。
5. 能否使用操作系统、框架或平台能力。
6. 能否使用项目已安装依赖。
7. 能否用一处小改动解决。
8. 以上都不行时，才新增最小实现。

## 硬规则

- 能删除就不要包装。
- 能复用就不要重写。
- 能显式分支就不要动态分发。
- 单实现不建抽象层。
- 单调用方不建通用框架。
- 单固定值不建配置系统。
- 不为“以后可能需要”写代码。
- 不引入新依赖，除非现有能力无法满足且收益超过安装成本。
- 修 bug 时改共享根因，不只修表面入口。
- 保留必要验证、错误处理和用户明确要求，不把偷懒伪装成简化。

## 强度

- `lite`：完成请求，并额外指出一个更简单方案。
- `full`：默认模式；执行决策阶梯，主动删减不必要实现。
- `ultra`：强质疑需求边界，先给删除清单，再考虑实现。
- `off`：关闭本 skill 的复杂度约束。

## 命令行为

- `/ponytail-review`：只看当前 diff，输出按优先级排序的删除清单。
- `/ponytail-audit`：看整个仓库，输出可删除抽象、重复实现和无收益复杂度。
- `/ponytail-debt`：记录本次没有立即处理的复杂度债务，包含文件、原因和后续动作。
- `/ponytail-gain`：报告已经量化的收益，例如少了多少代码、依赖、分支、配置或运行成本。
- `/ponytail-help`：列出命令和当前强度。

## 输出要求

- 实施类任务先改代码，再解释。
- 评审类任务先列问题，不写总览开场。
- 每条建议必须指向具体文件、符号或行为。
- 如果没有可删内容，明确说没有发现过度设计。
- 最终回复最多说明三件事：删了什么、复用了什么、没做什么。
