# Camera Pipeline Architecture

## 目标

`camera_pipeline` 是部署在 Linux 远端的运行时代码集合。其设计目标是让数据流动、算法计算、协议契约和调试产物清晰分离，避免把传输、采集、存储和算法混在同一个模块里。

## 核心原则

1. `pipeline_context` 是唯一负责数据交换、帧获取、资源管理和上下游编排的地方。
2. 其余子模块只做单一职责。
3. 子模块不得引入 `camera_pipeline` 之外的运行时依赖。
4. 子模块不得承担数据传输职责。
5. 子模块输入和输出优先使用协议对象，不使用跨层无结构 `dict` 作为主接口。
6. 除明确要求的 debug 信息外，不做额外 IO，不要求额外文件读写。
7. debug 可通过入参显式关闭。关闭时只计算关键结果，不构造 debug 产物。
8. 每个模块都应配套 `README.md`，并准确描述算法逻辑、输入、输出和 debug 字段格式。

## 模块边界

1. `pipeline_context`
   - 负责连接相机流、管理帧生命周期、缓存或转发必需资源。
   - 负责把上游输入转换成子模块可消费的协议对象。
   - 不实现具体算法。
2. 算法子模块
   - 只接收协议入参，输出协议结果。
   - 只实现本模块算法逻辑。
   - 不负责 RPC、ZMQ、文件系统、相机连接、远端转发、服务端口配置或启动管理。
3. 服务与适配层
   - 只做协议编解码、请求响应和上下文桥接。
   - 不把算法逻辑写进传输层。

## 输入输出约束

1. 传入数据必须以协议定义，字段语义、单位、坐标系都应明确。
2. 输出数据必须以协议定义，关键结果优先，debug 数据作为可选附加信息。
3. 需要对比或采集的先验信息，也应该以协议字段承载，不在算法内部隐式读取外部文件。
4. 若模块需要 debug，必须允许通过入参关闭 debug 计算。
5. 关闭 debug 时，不应计算或分配不必要的 debug 图像、掩码或大对象。

## README 要求

1. `camera_pipeline` 下每个新增或明显改动的子模块都应补充 README。
2. README 必须写清楚真实算法逻辑，不写营销式描述。
3. README 必须说明：
   - 输入协议
   - 输出协议
   - debug 协议
   - 坐标系约定
   - 先验如何参与计算
   - 关闭 debug 后哪些内容不再生成
4. 如果模块存在先验修正、坐标系叠加或多阶段位姿求解，README 必须写出矩阵关系和顺序。

## 评审检查点

1. 该模块是否只做单一职责。
2. 是否引入了 `camera_pipeline` 之外依赖。
3. 是否把数据传输、设备访问或文件 IO 混进算法文件。
4. 入参和出参是否都走协议。
5. debug 是否可关闭，关闭后是否还残留 debug 计算。
6. 是否存在 README，且 README 是否与真实逻辑一致。

## 工作方式

当需要修改 `camera_pipeline` 相关代码时，优先先检查：

1. 哪一层应该持有数据流动。
2. 哪一层应该持有算法。
3. 哪些数据应该通过协议表达。
4. 是否需要补 README。

如果发现新设计会打破上述边界，先修正边界，再写实现。
